# 公開版リポジトリ向け 実装仕様詳細設計（Codex再構成用）

## 1. 本ドキュメントの目的
本書は、現行リポジトリから新規公開リポジトリへ移植する際に、**Codexが迷わず再構成を実行できる実装仕様**を定義するための詳細設計書である。  
要件整理（`docs/公開向け再構築_要件整理.md`）を上位方針とし、本書は「実装時の判断基準・成果物定義・完了条件（DoD）」を提供する。

## 2. 適用範囲
- 対象: CTMC推定サロゲートモデルの公開版初期リリース（v0.1想定）
- 非対象:
  - 研究途中の派生モデル全量移植
  - 全Notebookの完全再現
  - 実データの再配布（法務確認前）

## 3. 移植対象ファイル（新規リポジトリへ持ち込むmd）
新規リポジトリでは、以下のmdを初期セットとして配置する。

1. `docs/公開向け再構築_要件整理.md`（方針・確認事項）
2. `docs/公開版_実装仕様詳細設計.md`（本書）
3. `README.md`（公開向けに再作成）
4. `CONTRIBUTING.md`（公開運用手順）

### 移植ルール
- 日本語原本を正とする（必要なら英語版は派生作成）。
- mdの見出しID互換のため、見出し階層は大きく変更しない。
- Codex入力に必要な要件（DoD・禁止事項・確認項目）を本文中に保持する。

### 禁止事項
- 日付命名ファイル（例: `model_0916.py`）の新規作成を禁止。
- Notebook内に本番ロジックを直接実装することを禁止。
- `__pycache__`, `.DS_Store`, 一時画像をGit追跡することを禁止。

## 6. 機能別 詳細設計（初期実装スコープ）

### 6.1 データ生成（`src/.../data`）
- 役割: CTMC仮想データ生成、分布パラメータ管理、シード固定生成
- インターフェース案:
  - `generate_dataset(config: dict, seed: int) -> Dataset`
- 検証観点:
  - 状態数整合性
  - 生成サンプル数一致
  - 分布範囲逸脱の検出

### 6.2 特徴量変換（`src/.../features`）
- 役割: 観測サンプルをDeepSets入力形式へ変換
- インターフェース案:
  - `build_features(samples, strategy) -> np.ndarray | torch.Tensor`
- 検証観点:
  - 入出力shape
  - batch処理時の順序不変性

### 6.3 モデル本体（`src/.../models`）
- 役割: サロゲート推定器定義
- インターフェース案:
  - `build_model(model_config) -> nn.Module`
- 検証観点:
  - パラメータ総数
  - 出力次元がQ推定要件を満たすこと

### 6.4 学習・評価（`src/.../train`, `src/.../eval`）
- 役割: 学習ループ、検証、指標出力
- インターフェース案:
  - `fit(model, train_loader, valid_loader, config) -> TrainResult`
  - `save_run_artifacts(run_dir, model, config, metrics)`
  - `evaluate(model, dataset, metrics) -> EvalReport`
- 学習時の必須仕様:
  - 学習開始時に `model_config.yaml` を `run_dir` へ自動保存
  - 学習終了時に `best_model.pt` と `metrics.json` を同一 `run_dir` へ保存
- 検証観点:
  - 学習の完走
  - 指標JSON出力
  - 可視化画像のアーティファクト保存

### 6.5 推論（`src/.../inference`）
- 役割: 学習済みモデルロードと推定
- インターフェース案:
  - `load_predictor_from_run_dir(run_dir) -> Predictor`
  - `predict(run_dir, input_data, output_mode="lifetime|raw") -> prediction`
- 推論時の必須仕様:
  - `run_dir` 配下の `model_config.yaml` と `best_model.pt` からモデルを自動再構築
  - CLIはモデル個別パスではなく `--run-dir` 指定を標準とする
  - 推論出力はデフォルトで寿命指標（逆数変換）を返しつつ、`output_mode=raw` で素の出力パラメータを取得可能にする
  - 設定は `run_dir` 内 `model_config.yaml` を最優先とし、CLI上書きは原則禁止とする
- 検証観点:
  - モデル読込失敗時のエラーメッセージ
  - 単発推論・バッチ推論の両対応

## 7. テスト方針（現時点の合意）

- ユーザー方針により、**初期の再構成フェーズではテスト実装を必須としない**。
- ただし、CLI実行可能性の確認（手動スモーク）は行う。
- 将来的に公開品質を引き上げる段階で、自動テスト導入を再検討する。

## 8. 移植作業の実行順序（新規リポジトリ側）
1. 基本骨格（ディレクトリ + `pyproject.toml` + `README`）作成
2. `docs/` に要件md 2本を移植
3. データ生成・特徴量変換から順に実装
4. 学習/評価CLI実装
5. （任意）テスト・CI導入（v0.2以降で再検討）
6. ベースライン結果固定（タグ）

## 9. 仕様未確定事項（実装前に要回答）
ユーザー要求の曖昧性を解消するため、以下を先に確定する。

1. 初回公開で対象にするモデルは「モデル1のみ」か「モデル1+モデル2」か。
2. Bayes系モデルを同梱するか。
3. 実データの公開可否（法務・許諾）。
4. 公式サポート環境（Python/GPU有無）。
5. 再現性レベル（R1/R2/R3）の必須ライン。

## 11. 追加で確定すべき要件（実装着手前の最終質問票）
以下は、現状ドキュメントに加えて実装手戻りを大きく減らすために確定しておきたい項目である。

### 11.1 データガバナンス
1. `real_data` の各ファイルごとの公開可否と根拠（契約・利用規約・社内規程）は明確か。
2. 二次配布不可データを含む場合、ダミーデータ生成スクリプトで代替する方針でよいか。
3. データの版管理単位（raw/interim/processed）と保存期間をどう定義するか。

### 11.2 ベンチマーク設計（確定）
1. 比較対象は **MLE** とする。
2. 主指標は **出力値の逆数（指数分布寿命の平均値）** とし、誤差は **MAE** で統一する。副次情報は **推論時間のみ** を記録する。
3. レポート集計は **データセット別 + 状態別** を基本とし、仮想データは **全体平均** も算出する。

### 11.3 運用要件（MLOps最小構成）
1. 実験トラッキング基盤（ローカルCSV / MLflow / W&B）を何にするか。
2. 学習済み重みの命名規則・保存先・保持世代数をどうするか。
3. 失敗ジョブ再実行時の仕様（resume可否、checkpoint間隔）を定義するか。

### 11.4 CLI/設定スキーマ
1. `configs/` の優先順位（デフォルト < 実験設定 < CLI上書き）を固定するか。
2. 乱数シード未指定時の振る舞い（エラー/自動採番）を統一するか。
3. すべてのCLIで `--dry-run`（設定検証のみ）を必須化するか。

### 11.5 品質・保守
1. 破壊的変更時のバージョニング規約（SemVer）を採用するか。
2. 公開後の問い合わせ窓口（Issueテンプレート、SLA目安）を整備するか。
3. テスト導入の再検討タイミング（例: v0.2以降）をいつにするか。

### 11.6 受け入れ基準（公開判定ゲート）
1. 「公開可」の判定条件を定量化するか（例: 指標閾値、手動スモーク成功、再現確認）。
2. CPU環境での最小実行パス（学習短縮版）を必須にするか。
3. READMEのクイックスタートがクリーン環境で再現できることをリリースゲートにするか。

---
上記質問への回答は、初回公開の実装スコープと運用負荷を直接左右するため、最初の設計確定ミーティングで合意することを推奨する。


## 12. 最終確認事項（確定）
実装着手前に、以下4点のみ合意したい。

1. **実行時間の定義**: 福祉表に記載する実行時間は **推論時間のみ** とする。
2. **寿命指標の誤差定義**: 逆数変換後の誤差は **MAEで統一** する。
3. **run_dir命名規則**: **連番運用で可**（例: `EXP-0001`, `EXP-0002`）。
4. **推論時設定優先順位**: **run_dir内のmodel_configを最優先** とする。
5. **推論時の出力柔軟性**: 寿命指標に加えて、必要時に **素の出力パラメータ** を取り出せるAPIを維持する。
