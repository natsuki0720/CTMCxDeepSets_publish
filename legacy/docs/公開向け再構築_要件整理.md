# 公開向け再構築に向けた要件整理（実装前）

## 目的
本ドキュメントは、現行リポジトリを「第三者が再現・検証・拡張しやすい公開版」に再構築するための、**実装前の要件確認**を目的とする。

## 現状から見える課題（公開観点）
1. **実験資産とプロダクト資産の混在**  
   Notebook、画像、CSV、重み、メモ、旧版モデルが同居しており、利用者が実行導線を把握しづらい。
2. **バージョン管理されていない実験履歴の混在**  
   `model_0715.py` のような日付命名が多く、どれが最新・推奨実装か判断しづらい。
3. **再現性に必要な定義不足**  
   依存関係・Pythonバージョン・実行手順・評価基準の機械可読な定義が不足している。
4. **公開運用に不向きなファイルの同梱**  
   `__pycache__`、`.DS_Store`、一時Notebookや生成画像が多く、配布物としてノイズが大きい。
5. **データ・学習済み重みの取り扱い方針未整理**  
   公開可否、ライセンス、配布形式（同梱 or 別ダウンロード）が不明確。

## 新規公開リポジトリで再実装・再構成すべき内容（案）

### A. リポジトリ構成の再設計（最優先）
- `src/`（ライブラリ本体）、`scripts/`（実行エントリ）、`configs/`（実験設定）、`tests/`（自動テスト）、`docs/`（理論と使い方）に分離。
- Notebookは `notebooks/` に限定し、論文図表再現用途に限定（本番実行フローから分離）。
- 旧版モデルは公開版では削除し、必要なら `archive/` へ移管。

#### Aの詳細要件
- **推奨ディレクトリ雛形**
  - `src/ctmc_surrogate/`
    - `data/`: データ生成・読み込み
    - `features/`: 行列/系列からモデル入力への変換
    - `models/`: ネットワーク定義
    - `train/`: 学習・検証ループ
    - `eval/`: 評価指標・可視化ロジック
    - `inference/`: 学習済みモデル推論API
  - `scripts/`
    - `train.py`, `evaluate.py`, `predict.py`, `prepare_data.py`
  - `configs/`
    - `dataset/*.yaml`, `model/*.yaml`, `train/*.yaml`, `experiment/*.yaml`
- **命名規則**
  - 日付付きファイル名（例: `model_0916.py`）を廃止し、責務ベース命名に統一（例: `deepsets_regressor.py`）。
- **アセット管理**
  - 画像・中間CSVは `artifacts/` へ出力し、Git管理対象外を基本とする。
  - 再現に必要な最小成果物のみ `docs/assets/` に固定配置する。

### B. API設計と責務分離
- 以下をモジュール責務として分離して再実装：
  - CTMCデータ生成
  - 尤度計算（MLE側）
  - 特徴量変換（行列→モデル入力）
  - 学習ループ
  - 推論・評価
- 研究用コード（検証）と推論用コード（利用）を分離。

#### Bの詳細要件
- **公開API（利用者向け）**
  - `fit(config) -> model`
  - `predict(model, samples) -> q_hat`
  - `evaluate(model, dataset, metrics) -> report`
  - `estimate_mle(samples, options) -> q_mle`（ベースライン比較用）
- **内部API（開発者向け）**
  - `generate_ctmc_dataset(params, seed)`
  - `build_features(samples, strategy)`
  - `compute_ctmc_loglikelihood(q, samples)`
- **境界定義**
  - `src/` は副作用最小（純粋関数中心）、I/O は `scripts/` 側へ寄せる。
  - Notebookは `src/` のAPIを呼ぶのみとし、独自実装を禁止する。
- **エラーハンドリング方針**
  - 入力検証エラー（shape不一致、状態数不整合）は `ValueError` 系で明示。
  - 乱数・計算失敗時は再実行可能なメッセージを付加する。

### C. 実験再現性の担保
- 依存関係管理（`pyproject.toml` など）を導入。
- 乱数シード、データ生成条件、学習ハイパーパラメータを設定ファイル化。
- 学習・評価コマンドを固定し、READMEから1コマンドで実行可能にする。

#### Cの詳細要件
- **再現実行レベル定義**
  - `R1`: 学習が完走し指標レンジが一致
  - `R2`: 主要図表が同様傾向で再現
  - `R3`: 乱数シード固定で数値がほぼ一致（許容誤差指定）
- **固定すべき要素**
  - Python / ライブラリバージョン
  - 乱数シード（Python / NumPy / PyTorch）
  - データ生成分布、サンプルサイズ、学習エポック、最適化手法
- **設定運用**
  - 1実験=1設定ファイル+1出力ディレクトリで管理。
  - 実験IDを採番し、設定・ログ・指標・図を同一IDで紐づける。
- **実行コマンド例（仕様案）**
  - `python scripts/train.py --config configs/experiment/uniform_baseline.yaml`
  - `python scripts/evaluate.py --run-id EXP-2026-0001`

### D. テスト・品質保証

#### Eの詳細要件

  5. 既知の制約（適用可能な観測間隔分布、長寿命領域での注意点）
- **ドキュメント分割**

## 仕様確定メモ（ユーザー指定反映）
- 比較対象は **MLE** とする。
- 主指標は **出力値の逆数（指数分布寿命の平均値）** とし、誤差は **MAE** で統一する。副次情報は **推論時間のみ** を記録する。
- レポート集計は **データセット別 + 状態別** を基本とし、仮想データでは **全体平均** も算出する。
- 学習時に Model Config を自動保存し、推論時はディレクトリパス指定からモデルを再構築する（Config優先）。
- 初期再構成フェーズでは、テスト実装は必須にしない。
- `run_dir` 命名は連番運用で良い（例: `EXP-0001`）。
- 推論時は寿命指標に加えて素の出力パラメータも取得可能にする。
